§1 System Overview

Problem Statement

Ingest high-velocity event streams from heterogeneous producers, apply stateful aggregations (counts, sums, percentiles, distinct-counts) across configurable time windows, and serve strongly-consistent query results under mid-scale load.

Key Challenges

  • Strong consistency across distributed aggregators
  • Bounded memory for stateful windowed reductions
  • Tail-latency spikes from stragglers
  • Exactly-once semantics for idempotent aggregates
  • Distributed tracing across async pipeline stages

§5 MapReduce & Dremel Aggregation Patterns

Map Phase — Key Extraction

// Each mapper reads from a Kafka partition
func mapEvent(raw []byte) (MappedEvent, error) {
  evt := parse(raw)
  key := AggKey{
    MetricName: evt.Name,
    TenantID:   evt.TenantID,
    Window:     truncate(evt.Ts, 5*time.Minute),
  }
  return MappedEvent{Key: key, Value: evt.Value}, nil
}

Reduce Phase — Combine & Commit

// Aggregator holds window state in memory,
// flushes via Raft on window close
type WindowAccumulator struct {
  Count  int64
  Sum    float64
  HLL    *hyperloglog.Sketch // distinct count
  DDSketch *ddsketch.DDSketch // percentiles
}

func (a *WindowAccumulator) Merge(v float64) {
  a.Count++; a.Sum += v
  a.HLL.Insert(v); a.DDSketch.Add(v)
}

Dremel-style Columnar Query (Cold Tier)

Google Dremel encodes nested + repeated fields using definition/repetition levels, enabling column projection without materializing full records. We apply the same model for cold-tier Parquet files.
-- Dremel-style nested query over Parquet
SELECT
  window_start,
  tenant_id,
  metric_name,
  SUM(value.sum)        AS total_sum,
  SUM(value.count)      AS total_count,
  APPROX_PERCENTILE(value.sketch, 0.99) AS p99
FROM agg_cold
WHERE window_start BETWEEN '2024-01-01' AND '2024-01-31'
GROUP BY 1, 2, 3
ORDER BY total_sum DESC
LIMIT 100;

-- Column pruning: only value.sum, value.count, value.sketch
-- are read from disk — not the full row group

§6 Distributed Consistency — Raft Protocol

Target: Linearizability (strongest single-object consistency). Every read reflects the most recent completed write, regardless of which replica is queried.

Write Path (Raft)

  1. Reducer receives aggregation update
  2. Leader appends to Raft log
  3. Leader replicates to ≥2 followers (quorum=3)
  4. Once majority ACK, entry is committed
  5. Leader applies to in-memory state + Redis hot tier
  6. ACK returned to producer ← linearizable point

Read Path (Linearizable)

  1. Query engine sends read to Raft leader
  2. Leader issues ReadIndex to confirm it is still leader
  3. If ≥ quorum confirms leadership → serve from state machine
  4. Otherwise: redirect to new leader
  5. Hot-tier Redis reads are served only after leader confirmation
Stale reads from follower replicas are explicitly disabled. All reads route to current Raft leader.

Exactly-Once Aggregation

// Each event carries a globally-unique idempotency key
// Reducer deduplicates within window using a Bloom filter
type IdempotencyKey = (ProducerID, SequenceNum)

func (r *Reducer) Apply(evt Event) error {
  if r.seen.Contains(evt.IdemKey) {
    return nil // duplicate — skip
  }
  r.acc.Merge(evt.Value)
  r.seen.Add(evt.IdemKey)
  return r.raft.Propose(r.acc) // commit via Raft
}

§7 Distributed Tracing — Dapper Model

Based on Google's Dapper paper: every request is tagged with a TraceID (globally unique). Each pipeline stage creates a Span with a SpanID and ParentSpanID, recording start/end timestamps and annotations. Spans are sampled and collected asynchronously — zero overhead on the hot path when sampling rate = 0.

Trace Context Propagation

// W3C Trace Context (traceparent header)
traceparent: 00-4bf92f3577b34da6...-00f067aa0ba902b7-01
              ↑  ←— TraceID (128-bit) —→  ←ParentSpan→  flags

// Each stage extracts + re-injects context
func processEvent(ctx context.Context, evt Event) {
  span := tracer.Start(ctx, "aggregator.apply")
  defer span.End()

  span.SetAttributes(
    attribute.String("agg.key",    evt.Key),
    attribute.String("agg.window", evt.Window),
    attribute.Int64 ("agg.shard",  evt.Shard),
  )
  applyToAccumulator(ctx, evt)
}

§8 Tail-Latency Controls

At 100K req/s, a 1% straggler rate = 1,000 slow requests/sec. Tail-latency techniques below keep p999 within 3× of p50.

1. Hedged Requests

If a shard doesn't respond within the 95th-percentile latency, send the same request to a backup shard and use whichever replies first. Cancel the loser.

func hedgedQuery(key AggKey) (Result, error) {
  ctx, cancel := context.WithTimeout(ctx, 50*ms)
  defer cancel()

  // Issue primary, hedge after p95 deadline
  hedge := time.After(p95Latency)
  select {
  case r := <-queryPrimary(ctx, key):
    return r, nil
  case <-hedge:
    return race(ctx, key, primary, backup)
  }
}

2. Speculative Execution

For Dremel-style fan-out queries, detect slow sub-tasks and re-launch them on idle workers. Use the first completion, discard duplicates.

func speculativeReduce(shards []Shard) Result {
  results := make(chan ShardResult, len(shards))

  for _, s := range shards {
    go queryWithSpeculation(s, results)
  }
  // Collect all; speculative copies deduped
  return mergeFirst(results, len(shards))
}

3. Timeout Budgets (Cascading)

// Deadline propagated via context — each hop
// subtracts its own timeout from remaining budget

4. Request Coalescing (Write Batching)

// Batch small writes into micro-batches
// to amortize Raft round-trip overhead
func (r *Reducer) RunBatcher() {
  ticker := time.NewTicker(5 * time.Millisecond)
  for {
    batch := r.drainQueue(512) // max 512 events
    if len(batch) > 0 {
      applyBatch(batch)
      r.raft.Propose(r.acc)  // single Raft entry
    }
    <-ticker.C
  }
}

§9 Architecture Decision Records

ADR-001 Strong consistency via Raft over eventual consistency Accepted
Context
Aggregated metrics are used for billing, SLA reporting, and alerting — stale reads could trigger incorrect charge-backs or missed alerts.
Decision
Use Raft-based aggregator cluster (etcd / TiKV) with linearizable reads via ReadIndex. All reads route to the Raft leader.
Consequences

Higher write latency (~15ms Raft round-trip) vs eventual consistency. Leader becomes a bottleneck — mitigated by per-key sharding (each key-range has its own Raft group). Follower reads disabled: all reads go to leader.

ADR-002 Micro-batch aggregation (5ms windows) over pure streaming Accepted
Context
Pure streaming (per-event Raft propose) would require 100K Raft writes/sec — unsustainable. Pure batch (1m flush) adds unacceptable latency.
Decision
Buffer events in memory for 5ms, apply all to accumulator, then issue a single Raft proposal. Reduces Raft ops to ~200/sec at 100K events/sec.
Consequences

+5ms write latency (within budget). Memory buffer must survive crashes — covered by Kafka offset not advancing until Raft commit. Accumulator state is in-memory only; crash before commit = replay from last Kafka offset.

ADR-003 Dremel-style columnar Parquet for cold storage Accepted
Context
Historical aggregation queries span months of data across billions of rows. Row-based formats are too slow; wide columnar formats can't represent nested event schemas.
Decision
Encode nested aggregation records as Parquet with definition/repetition levels (Dremel encoding). Query via DuckDB / Athena with column-pruned scans.
Consequences

10–100× faster analytical scans vs row-based. Write path must serialize accumulators to Dremel-encoded Parquet on window close. Schema evolution handled via Avro compatibility rules before Parquet write.

ADR-004 Hedged requests for tail-latency control at query layer Accepted
Context
GC pauses, disk I/O spikes, and kernel scheduling cause ~1% of queries to exceed 3× median latency. Retries add full latency penalty.
Decision
Issue hedged duplicate query to a second shard after p95 deadline. First response wins; loser is cancelled. Adds ~5% extra query load.
Consequences

p999 read latency drops from ~200ms to <60ms in load tests. Must ensure hedged queries are idempotent (reads are). Hedging disabled for writes — Raft handles write safety.

ADR-005 Adaptive hybrid sampling for Dapper-style tracing Accepted
Context
100% trace sampling at 100K events/sec generates ~500 MB/s of span data — prohibitively expensive to store and query.
Decision
Head-based 1% probabilistic sampling for normal traffic. Tail-based always-sample for errors and latency > p95. Span export is async via OTLP gRPC — zero blocking on the hot path.
Consequences

~1K sampled traces/sec at steady state + all slow/error traces. Storage ~5 MB/s (manageable). Root-cause investigation possible for any error or slow trace. Normal-traffic trace coverage is statistical (1%).
